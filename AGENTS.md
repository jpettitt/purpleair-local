# Coding agent instructions

This file tells AI coding agents (Claude Code, Copilot, Cursor, Cody,
and similar) how to contribute to this repository. Read it before
making any change. It supersedes any default behaviour baked into the
tool.

A short [`CLAUDE.md`](CLAUDE.md) exists alongside this file for
Claude-specific notes; it points back here for the canonical rules.
Human contributors should start with [`CONTRIBUTING.md`](CONTRIBUTING.md).

## What this project is

**PurpleAir Local** is a Home Assistant custom integration that polls
PurpleAir PA-II (and compatible) sensors directly over the LAN. No
PurpleAir cloud, no API key. Python only — no frontend code in this
repo. Integration code lives under
[`custom_components/purpleair_local/`](custom_components/purpleair_local/);
tests under [`tests/`](tests/).

For project context, design rationale, and the roadmap, read in this
order:

- [`README.md`](README.md) — what the integration does, install
  instructions, dashboard examples for the AQI category attributes.
- [`DESIGN.md`](DESIGN.md) — architecture, the firmware-vs-docs
  discrepancies the parser tolerates, the AQI correction formulas, the
  channel-disagreement primary-fallback rule, and the breakpoint table
  rationale.
- [`TODO.md`](TODO.md) — the working punch list. Shipped items stay
  here as a historical record with branch / test notes; open items are
  at the bottom under "Post-v0.1.0".
- No `CHANGELOG.md`. Per-release narrative lives in
  [GitHub Releases](https://github.com/jpettitt/purpleair-local/releases);
  the cut-a-release procedure is below.

## Authority and limits

**Agents open pull requests. Humans review and merge.** No
self-merging, no direct commits to `main`, no merges via the GitHub
API. The `main` branch ruleset enforces this (it requires CodeQL to
pass and has admin-bypass enabled for emergencies only — admin bypass
is a human's tool, not an agent's).

This is non-negotiable, even for trivial changes. The "trivial change
that breaks production" failure mode is real, and the human review
step is the cheapest insurance against it.

### Hard no's — never do these without explicit user approval in the same session

1. **History rewrites.** `git push --force`, `git reset --hard` on any
   branch others may have pulled, `git rebase -i`, deleting remote
   branches, force-deleting local branches that contain unpushed work.
2. **Version bumps, tags, or releases.** Do not edit `version` in
   [`custom_components/purpleair_local/manifest.json`](custom_components/purpleair_local/manifest.json).
   Do not create tags. Do not `gh release create`. Releases are always
   human-initiated and follow the procedure in the "Cutting a release"
   section below.
3. **CI / packaging / dependencies.** Do not modify anything under
   [`.github/workflows/`](.github/workflows/),
   [`hacs.json`](hacs.json),
   [`requirements_test.txt`](requirements_test.txt),
   [`pyproject.toml`](pyproject.toml), or any lockfile. Suggest changes
   in the PR description and let a human apply them. (Note: pyproject
   here is deliberately *not* a `[project]` block — adding one breaks
   HA's loader on Python 3.14, see
   [`pyproject.toml`](pyproject.toml) for the rationale.)
4. **Governance files.** Do not edit this file (`AGENTS.md`),
   [`CLAUDE.md`](CLAUDE.md), [`CONTRIBUTING.md`](CONTRIBUTING.md),
   [`.github/PULL_REQUEST_TEMPLATE.md`](.github/PULL_REQUEST_TEMPLATE.md),
   or [`LICENSE`](LICENSE). Suggest revisions in the PR body and let a
   human apply them.
5. **Dev testbed infrastructure.** Do not edit
   [`docker-compose.yml`](docker-compose.yml), the [`Makefile`](Makefile),
   or [`.dev/configuration.yaml.example`](.dev/configuration.yaml.example).
   These are tuned for the maintainer's local dev loop.
6. **Test fixtures.** Do not edit
   [`tests/fixtures/`](tests/fixtures/). These are redacted real-sensor
   captures and serve as the regression-test ground truth. If a fixture
   needs updating (firmware upgrade changed a field, new field
   appeared) raise it in the PR description and let a human
   re-capture and re-redact.
7. **Deletions.** Do not delete files or directories. Renames are fine.
   If a file looks unused, raise it in the PR description.

If a task seems to require any of the above, stop and ask. Do not
work around the rule by structuring the change differently.

## The diagnostic discipline

**No fixes without understanding.** This is the single most
load-bearing rule in this repo.

- Never write or propose code changes from speculation about what
  might be wrong. Diagnose first; *then* propose a fix.
- Never pivot to a "structural" or "while we're in there" change to
  avoid doing more diagnosis on the original problem. If the current
  diagnostic isn't giving you enough signal, improve the diagnostic —
  add a log line, add a probe, hit the real device — rather than
  guessing.
- Never proceed past a diagnostic step without explicit user
  go-ahead. Surface what the data shows, propose what to change, wait.
- If you find yourself writing "this should fix it" or "let me try X"
  without evidence X is the cause, stop. Read the rule again.

This applies equally to fresh problems and to fixes that don't work
on the first try. A failed fix is not a license to start trying
things — it's a signal that the diagnosis was wrong.

### Worked example — the rule behind a specific rule

This integration's parser tolerates field-name and value-spelling
quirks that the upstream PurpleAir community docs do not document:

- The on-device PM2.5 AQI field is `pm2.5_aqi` (literal dot in the
  JSON key) in firmware 7.02. The docs say `pm2_5_aqi`.
- The `place` field returns `inside` / `outside`. The docs say
  `indoor` / `outdoor`.
- Several fields (`pa_latency`, `latency`, `status_7`) appear in real
  payloads but aren't in the docs at all.

If we had trusted the docs and coded against `pm2_5_aqi` /
`indoor`/`outdoor`, the parser would silently return `None` for AQI on
every real device. Instead, we probed both real sensors first, diffed
the actual JSON against the docs, then coded to the firmware. The
parser accepts both spellings so future firmware that aligns with the
docs still works. See the parser tests in
[`tests/test_models.py`](tests/test_models.py) and the fixtures in
[`tests/fixtures/`](tests/fixtures/).

The takeaway is general: **for any device, parse the wire format the
device actually emits, not the format the docs claim.** The docs are
a hint; the device is the source of truth.

## Workflow

### Branching

New work on a new branch off `main`. Naming:

- `issue-<number>-<short-description>` when addressing a tracked issue
  (e.g. `issue-12-aqi-stale-on-reboot`).
- `feature/<short-name>` for new functionality
  (e.g. `feature/zeroconf-discovery`).
- `fix/<short-name>` for bug fixes
  (e.g. `fix/parser-handles-empty-cf1`).
- `docs/<short-name>` for documentation-only changes.
- `release/v<x.y.z>` for release commits (humans only — see "Cutting
  a release").

Don't reuse a stale branch for unrelated work. Don't push to someone
else's branch without arrangement.

### Pre-commit gate — must pass before any commit

The repo has one local test gate:

```bash
make test    # equivalent to .venv/bin/pytest
```

All tests must pass. Net new tests must not reduce coverage of any
existing module's branches; bug fixes must come with a regression
test that fails on `main` and passes with the fix.

CI runs additional checks that you cannot easily reproduce locally
without standing up the HA test harness fully — they fire on every
push to a PR branch:

- **hassfest** — HA's integration manifest / schema / translations
  validator. Common failures: manifest keys not alphabetized after
  `domain` and `name`; a `strings.json` error key that has no matching
  code reference.
- **HACS validate** — HACS install validator. The `ignore: brands`
  config is intentional and permanent (see
  [`.github/workflows/validate.yml`](.github/workflows/validate.yml));
  custom integrations ship brand assets in-tree under
  [`custom_components/purpleair_local/brand/`](custom_components/purpleair_local/brand/)
  per the Feb 2026 Brands Proxy API change.
- **CodeQL / Analyze (python)** — GitHub default-setup code scanning.
  Skips PRs that touch no Python; for those, ask a human to merge with
  admin bypass.

If you modified only documentation (`*.md`, README assets,
`tests/fixtures/README.md`), you may skip `make test`, but you must
still verify the docs render by reading them back. Mark the PR test
plan "n/a — docs only".

### Setup, the first time

```bash
python3 -m venv .venv
.venv/bin/pip install -r requirements_test.txt
make test                                  # confirm baseline is green
```

### Adding tests

Every behavioural change must come with a test that fails on `main`
and passes with the change. Bug fixes get a regression test that
reproduces the original symptom. New features get tests covering the
golden path and the obvious edge cases (None inputs, missing
hardware, single-laser vs dual-laser sensors, etc.).

Layout:

- All tests under [`tests/`](tests/). Use `pytest`, follow existing
  patterns. Type-annotate test parameters. Prefer
  `pytest.mark.parametrize` over duplicated bodies. No conditional
  logic inside tests.
- Fixture-driven tests load from [`tests/fixtures/`](tests/fixtures/);
  synthetic-payload tests build minimal dicts inline via helpers like
  `_dual_payload()` in
  [`tests/test_sensor.py`](tests/test_sensor.py).
- Coordinator / config-flow / options-flow tests use the `hass`
  fixture from `pytest_homeassistant_custom_component`. Tests that
  need real sockets (the `api.py` aiohttp tests) opt back in with an
  autouse `socket_enabled` fixture — pytest-socket blocks by default
  via the HA plugin.

### Commit messages

One-line subject in the form `<type>: <summary>` or
`<scope>: <summary>`. Examples from this repo's history:

```
api.py: async client for sensor /json with retry + error taxonomy
models.py: parse /json into a typed SensorReading tree
aqi.py: EPA / AQandU / LRAPA corrections + 2024 EPA AQI table
review fixes: raw-AQI consistency, VOC class, manifest, README
ci: alphabetize manifest keys, ship in-tree brand assets, ignore brands
readme: dashboard examples for the new AQI color attributes
chore: bump manifest version to 0.1.0 for first release
```

Subject under ~72 characters. Body explains the *why* when the change
is non-obvious — never just restates the diff. Sign your work with a
`Co-Authored-By:` trailer when an agent contributed.

Do not amend or squash a commit once you have pushed it to a PR
branch. Reviewers track changes between reviews via the commit
history; rewriting that history erases context.

### Pull requests

When the work is ready:

1. Push the branch.
2. Open the PR against `main`. The repo's PR template at
   [`.github/PULL_REQUEST_TEMPLATE.md`](.github/PULL_REQUEST_TEMPLATE.md)
   auto-populates the body. Fill in every section; never remove one.
   Mark sections "n/a" with a one-line reason when they genuinely
   don't apply.
3. PR title follows the commit-subject conventions above.
4. The template's sections:
   - **Summary** — 1–3 bullets on what changed and why.
   - **Test plan** — what you ran locally and what passed. Code PRs
     must check the `make test` box. Docs-only PRs write "n/a — docs
     only" below the checklist.
   - **Risk** — anything a reviewer should think hardest about.
     Parser changes, anything that touches the entity tree shape, AQI
     correction math, channel-disagreement logic. "Low — internal
     refactor with full test coverage" is a valid answer when true.
   - **Docs touched** — tick the docs updated, or "n/a — internal
     change".
5. **Stop.** Do not self-merge, do not request reviewer dismissal, do
   not edit the PR after pushing unless responding to feedback. Do not
   use `gh pr merge --admin` even if you have the credential.

CI will run the gates listed above. If CI reports a failure you
didn't see locally, fix the underlying issue and push a new commit —
do not amend the failing one (see commit rule above).

## Code conventions

This project is built on top of Home Assistant. PRs will be reviewed
against Home Assistant's published coding standards, not a separate
project house style. The canonical document:

- [Home Assistant core AGENTS.md][ha-core-agents] — Python style,
  testing patterns, comment philosophy ("explain *why* — never what"),
  Python 3.13+ syntax expectations, type-hint expectations,
  async-first defaults.

Follow it. PR review will flag deviations.

### A worked example — pick the right HA primitive

When a Home Assistant primitive exists, use it; don't invent a
parallel. The cost of getting this wrong is silent UI breakage that
unit tests don't catch.

Concrete case from this repo's history: the VOC-resistance entity
originally declared `device_class=SensorDeviceClass.AQI` with a
trailing `# closest match` comment. AQI is a unitless index 0–500;
gas-sensor resistance is in ohms. HA's frontend trusts `device_class`
to drive unit conversion, formatting, and history-graph styling — so
the "closest match" choice would have caused the VOC entity to render
with AQI categories applied to an ohms value. The fix was to ship
without a `device_class` (HA has no good class for gas resistance)
rather than reach for a vaguely-similar one.

The general rule: **a `# closest match` or `# TODO: pick a better
device_class later` comment is a code smell, not an acceptable
compromise.** Either find the right HA primitive, or omit the
declaration so HA falls back to its generic handling.

## Documentation obligations

When a change touches user-visible behaviour or future plans, update
the relevant doc *in the same PR*. PR review checks for this; missing
doc updates are a request-changes item.

- **[`README.md`](README.md)** — installation, what the integration
  does, the "What it provides" entity catalogue, dashboard examples.
  Update on every user-facing feature change.
- **[`TODO.md`](TODO.md)** — when you finish a planned item, append an
  italicised "_Done on `<branch>`: …_" note to its entry rather than
  deleting the item. Open items at the bottom under "Post-v0.1.0";
  move new follow-ups there as you discover them.
- **[`DESIGN.md`](DESIGN.md)** — update when the architectural
  approach changes, when a new firmware quirk comes to light, or when
  a previously-documented decision is reversed. The AQI breakpoint
  table, the primary-channel fallback rule, and the
  firmware-vs-docs discrepancy list all live here.
- **GitHub Release notes** — human-only. Agents don't draft these; if
  a feature ships in a PR, the human cutting the next release will
  pull from the PR descriptions when they write the notes.

## Cutting a release (human-only)

This is here so you know the process exists, not because you should
run it. Agents stop at "PR open"; humans do the rest.

1. Merge the feature PRs that will be in the release.
2. New branch `release/v<x.y.z>` off `main`, bump
   `version` in `manifest.json`, push, open PR.
3. Wait for CI green. CodeQL may skip on a version-only PR; if it
   does, the admin merge bypass on the `main` ruleset is the
   workaround (see the rationale on the bypass actor in
   `.github/workflows/validate.yml` and the ruleset config).
4. After merge, tag `v<x.y.z>` on the merge commit, push the tag.
5. `gh release create v<x.y.z> --notes-file <notes>` with a real
   feature summary (not just the auto-generated changelog link).
6. Add the version + ship date to the open TODO entry in a follow-up
   PR if relevant.
